# 一致性hash

## 哈希环

分布式下的负载均衡之基石

### 解决问题 
如果有 3 台机器， 取模， 所有 key 经过 hash 之后， % 3 才能得到这个值。如果 2 挂掉了，**几乎** 所有 key 都失效了，雪崩。 如何减少这个动静？

构建一个环， 这个换上有 2^32 - 1 个槽位，*其实只要够大即可*，然后每个节点，根据其 IP 也好，mac 地址也好，只要唯一标识都行，进行 hash， 依次锁定到 [0-N1), [N1-N2), [N2-N3) 这几个区间。

一个 Key， 进行 Hash 的到 h1
1. 如果 0 <= h1 < N1, 则就在 1 中； 
2. 若 N1 <= h1 < N2, 则就在 2 中；
3. 若 N2 <= h1 < N3, 则就在 3 中;
4. 若 N3 <= h1, 则还找 1

这样就形成了一个闭环，即所有 key ， 一定在一个范围内。

#### 顺时针

其实大于 N3 后，其实可以视为 0. 因为一定在环上找到一个点，N1、N2、N3 就是一个标识，完整写法应该是 [N3, N1)。
当大于 N3 后，默认置于 N1 这个标识处。 

### 新问题

若 N1 = 100， N2 = 110， N3 = 1000000， 则会极度不平衡。而且还是 hash 算法算出来的范围，正常手段认为无法控制。

### 虚拟机

通过虚拟一些唯一键，创建更多节点：

> node1#1, node1#2, node1#3....node1#200

这样虽然是 3台机器，但我们有 600 个节点，虽说还有可能不均衡，但相对 3 来说， 已经均衡多了。

也就是说，通过加一个 map，多一次 O(1) 的运算，平均了这些值。

## QA

- Q. 如果一个机器性能高， 其他性能相对低，怎么搞？
    > A. 那这个机器多些虚拟节点
- Q. 一台挂掉 （减少机器）
    > A. 是不是只影响了它自己上面的 200 个虚拟节点的数据？ 相对 %3 ， 几乎雪崩的情况，改善了很多。 一致性 hash 么，同一个 key 每次都一样，也只有那 200 个虚拟节点的 key 会失效，其他机器上的 key 不受影响
- Q. 新加机器
    > A. 计算 node4 的 hash 范围， 假设它介于 N3 和 N1 之间， 那也只会影响到  N3 <= h < N4 的数据，N4 -> N1 之间的都影响不到 

它不是银弹，完全解决了问题，只是提高了可用性，减少了数据迁移量，而且这个数据只能是缓存， 不是永久的 key。
